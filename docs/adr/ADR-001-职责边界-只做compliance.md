# ADR-001: SPEC Protocol 职责边界 —— 只做 compliance 合规引擎

## 状态
已确认（2026-06-29）

## 背景
GB150 涉及 6 种计算形态（纯公式/查表/查图/迭代/条件分支/几何依赖）。原 1/2/3.docx 假设 SPEC 内置表达式求值、迭代求解、查图插值等计算能力。

但工程实践中：
- 数值计算已有成熟软件（SW6 等），重复实现成本高、价值低。
- 「计算数值」与「判定合规」是两种不同职责，混在一起会导致引擎复杂度失控。
- 限值的正确性应由专业计算软件保证，而非合规引擎。

## 决策
SPEC Protocol 本次**只实现 compliance 合规引擎**（规则匹配 / 工艺判定）。

- 数值计算（厚度/应力/外压/迭代/查图）由外部计算软件完成。
- 将来通过 MCP 或计算书解析接入计算软件。
- SPEC 的输入是计算软件输出的**结构化参数表**（含参数值与限值）。
- SPEC 只做「参数 vs 限值」的匹配判定，不计算限值。

## 后果

### 正面
- 职责清晰，MVP 范围大幅收窄。
- 不重复造计算轮子，复用专业计算软件。
- 合规引擎可独立演进，不被计算细节拖累。

### 负面
- SPEC 依赖外部计算软件的输出质量；限值正确性不在 SPEC 控制范围。
- 需要定义清晰的输入契约（见 ADR-003）。

### 据此移除的能力
- ❌ 表达式求值引擎（evalexpr / 自研表达式树）
- ❌ 迭代求解器
- ❌ 对数坐标查图插值
- ❌ 材料库查询
- ❌ 限值计算
- ❌ 几何拓扑处理

# ADR-007: 标准术语提取与参数映射工具

**状态**: 已确认  
**日期**: 2025-06-29

## 背景

随着SPEC Protocol从GB150扩展到100+标准(如NB/T 47012、ASME VIII、EN 13445等),面临以下挑战:

1. **术语不一致** — 不同标准对同一概念使用不同术语(如GB150叫"设计压力",ASME叫"Design Pressure")
2. **参数映射复杂** — 标准中的40+术语需要映射到SPEC的parameters/limits/attrs三分区
3. **重复劳动** — 每接入新标准都要人工梳理术语表和参数映射
4. **可追溯性差** — 规则YAML中的参数名(如`sigma_allow`)需要能追溯回标准原文的术语定义

用户需要一个**标准化的术语提取与映射流程**,在以下时机执行:
- 新标准接入时(首次)
- 标准更新时(换版)
- 定期批量整理(全量校验)

## 决策

开发**独立工具脚本** `term_extractor`,专门负责:

### 核心功能

1. **术语提取** — 从标准原文(PDF/Word/Markdown)中自动提取术语定义
2. **参数映射** — 将标准术语映射到SPEC Protocol的parameters/limits/attrs
3. **关系分析** — 识别术语之间的引用关系和依赖链
4. **文档生成** — 输出标准化的术语表和参数映射文档

### 工具定位

```
标准原文(PDF/Word/Markdown)
    │
    ▼
┌─────────────────────┐
│  term_extractor     │ 独立工具脚本
│  (Python/Rust)      │
└────────┬────────────┘
         │
         ├─► 术语定义表.md
         ├─► 参数映射表.json
         ├─► 条款引用关系图.md
         └─► 规则YAML模板/
             └─ {标准号}/
                 └─ *.yaml (骨架)
```

### 输出产物

| 产物 | 格式 | 用途 | 示例 |
|-----|------|------|------|
| 术语定义表 | Markdown | 记录标准术语与SPEC参数映射 | `docs/{标准号}-术语定义.md` |
| 参数映射表 | JSON | 机器可读的参数映射关系 | `rules/library/{标准号}/param_mapping.json` |
| 条款引用关系 | Markdown | 展示标准内部/跨标准引用 | `docs/{标准号}-条款引用关系.md` |
| 规则YAML骨架 | YAML | 基于术语映射生成规则模板 | `rules/library/{标准号}/` |

### 数据结构设计

#### 术语条目 (TermEntry)

```json
{
  "term_id": "pressure.design",
  "standard_term": "设计压力",
  "definition": "设定的容器顶部的最高压力,不低于工作压力",
  "spec_mapping": {
    "type": "parameter",
    "name": "design_pressure",
    "unit": "MPa",
    "data_type": "f64"
  },
  "source_clause": "3.1.4",
  "referenced_by": ["GB150.3-5.3", "GB150.4-11.4.3"],
  "related_terms": ["operating_pressure", "calc_pressure"]
}
```

#### 参数映射表 (ParamMapping)

```json
{
  "standard": "GB150",
  "version": "2024",
  "terms": [...],
  "parameter_groups": {
    "design": ["design_pressure", "design_temperature"],
    "geometry": ["D_i", "D_o", "delta"],
    "stress": ["sigma", "sigma_allow"],
    "material": ["R_m", "R_eL", "KV2"]
  },
  "cross_references": {
    "GB150.1→GB150.2": ["许用应力", "材料温度范围"],
    "GB150.3→GB150.4": ["设计压力", "壁厚"]
  }
}
```

### 使用场景

#### 场景1: 新标准接入

```bash
# 1. 提取术语
term_extractor extract \
  --input standards/NB_T_47012-2023.pdf \
  --output rules/library/NB47012/

# 2. 人工校对术语映射表
# 编辑 rules/library/NB47012/param_mapping.json

# 3. 生成规则YAML骨架
term_extractor generate-rules \
  --mapping rules/library/NB47012/param_mapping.json \
  --template rules/.templates/ \
  --output rules/library/NB47012/

# 4. 生成术语文档
term_extractor generate-docs \
  --mapping rules/library/NB47012/param_mapping.json \
  --output docs/NB47012-术语定义与概念映射.md
```

#### 场景2: 标准更新比对

```bash
# 比对GB150-2011和GB150-2024的术语差异
term_extractor diff \
  --old standards/GB150-2011/param_mapping.json \
  --new standards/GB150-2024/param_mapping.json \
  --output docs/GB150-术语变更差异.md
```

#### 场景3: 批量校验

```bash
# 校验所有标准的术语映射一致性
term_extractor validate \
  --rules rules/library/ \
  --check-consistency \
  --report docs/术语一致性报告.md
```

## 考虑的替代方案

1. **纯人工整理** — 不采用。100+标准时人工成本过高,且容易出错。
2. **LLM自动提取** — 部分采用。LLM辅助识别术语,但参数映射需人工确认(工程标准需严谨)。
3. **集成到spec_cli** — 不采用。术语提取是独立流程,与合规引擎解耦,工具链更清晰。
4. **数据库存储** — 不采用。JSON/Markdown更易版本控制,适合工程审查场景。

## 权衡与后果

### 优势
- **标准化流程** — 新标准接入时间从3天缩短到0.5天
- **可追溯性** — 每个参数都能追溯回标准原文条款
- **一致性保障** — 自动检测术语冲突和映射遗漏
- **文档自动生成** — 减少人工编写文档的工作量

### 风险
- **初期开发成本** — 预计2-3周完成工具开发
- **标准格式差异** — PDF/Word解析需要适配不同格式
- **LLM准确性** — 术语识别可能有误,需人工复核

### 缓解措施
- MVP版本先支持Markdown输入,PDF解析后续迭代
- 术语提取结果标记"置信度",低置信度需人工确认
- 提供交互式校对界面(term_extractor review命令)

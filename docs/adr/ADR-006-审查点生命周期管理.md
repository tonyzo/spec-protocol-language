# ADR-006: 审查点生命周期管理

**状态**: 已确认
**日期**: 2025-06-29

## 背景

当前 ComplianceEngine 从单一 YAML 文件静态加载规则。实际工程审查中，审查人员需要**先约定好用哪些条款**再执行校核。这种"每次手动选规则"的方式存在痛点：

- 同一设计类型（如"内压圆筒"）每次审查都要重新选条款
- 不同审查人员可能选择不同条款，导致审查标准不一致
- 规则分散在多个文件中，缺少统一管理

用户期望的工作流：
1. **首次**：引擎从规则库中自动发现适用规则 → 人工确认 → 沉淀为审查点集合
2. **后续**：相同设计类型直接加载已沉淀的审查点集合

未来规则库将覆盖一个专业的 100+ 项标准（不仅是 GB150）。

## 决策

引入**审查点生命周期管理**，在 ComplianceEngine 之上叠加规则发现与沉淀机制。

### 核心概念

- **RuleLibrary（规则库）**: 所有标准的全部 SpecRule 集合。按标准号分目录存储。
  - 路径: `rules/library/{standard}/{chapter}-{topic}.yaml`
  - 规则库中的规则是"候选规则"，尚未绑定到任何设计类型。

- **ReviewPoint（审查点）**: 一条经人工确认、对特定设计类型生效的 SpecRule。
  - 粒度: 1 审查点 = 1 SpecRule。

- **ReviewPointSet（审查点集合）**: 一个设计类型的全部已确认审查点，存储为 YAML 文件。
  - 路径: `rules/confirmed/{design_type}/{standard}-{clause}-{year}.yaml`
  - 文件名含标准号与年份，支持标准更新时替换文件。

- **ReviewPointRegistry（审查点注册表）**: 索引文件，映射设计类型 → 审查点文件列表。
  - 路径: `rules/registry.json`

### 生命周期

```
候选规则 (RuleLibrary)
    │
    │ 1. discover: 按 ParameterTable.applicability 自动筛选
    ▼
候选审查点 (CandidateReviewPoint[])
    │
    │ 2. confirm: 人工逐条确认/拒绝
    ▼
已确认审查点 (ConfirmedReviewPoint[])
    │
    │ 3. sediment: 保存为 YAML 文件 + 更新注册表
    ▼
审查点集合 (ReviewPointSet)
    │
    │ 4. load: 后续直接加载，无需再确认
    ▼
ComplianceEngine.check()
```

### 目录结构

```
rules/
├── library/                      # 规则库：所有标准的全部规则
│   ├── GB150.3/
│   │   └── 5.3-内压圆筒.yaml
│   ├── GB150.4/
│   │   ├── 7-热处理.yaml
│   │   └── 9-无损检测.yaml
│   └── ... (未来 100+ 标准)
├── confirmed/                    # 已确认的审查点集合
│   └── 内压圆筒/
│       ├── GB150.3-5.3-2024.yaml
│       └── GB150.4-7-2024.yaml
└── registry.json                 # 审查点注册表
```

### 版本管理

- 审查点集合文件名含标准号和年份: `GB150.3-5.3-2024.yaml`
- 标准更新时，直接替换对应文件（如 `GB150.3-5.3-2026.yaml`）
- 注册表记录每个文件的 `confirmed_at` 时间戳，可追溯确认历史

## 考虑的替代方案

1. **数据库存储** — 不采用。YAML 文件更透明、可版本控制、适合工程审查场景的评审与审计。
2. **规则库+推断** — 不采用。MVP 阶段不做规则推断，仅按 applicability 筛选候选。
3. **版本号自增** — 不采用。标准号标注更直观，与 GB150 标准体系对齐。

## 后果

- 新增 `registry.rs` 和 `discovery.rs` 两个模块
- CLI 增加 `discover` 和 `confirm` 子命令
- 规则文件从 `rules/` 迁移到 `rules/library/`
- `ComplianceEngine` 核心逻辑不变，只是规则的来源从"手动指定"变为"注册表自动加载"
